# 🎨 디자인 시스템 및 UI 가이드라인

본 문서는 앱의 사용자 경험(UX)과 시각적 브랜딩(UI)의 일관성을 확보하고, 모바일 환경에서의 조작 편의성을 위한 설계 규칙을 정의합니다.

---

## 1. 서체 (Font Family)

*   **기본 폰트**: **스포카 한 산스 네오 (Spoqa Han Sans Neo)**
*   **선정 이유**: 
    - 국문과 영문, 특히 숫자가 섞여 있을 때의 자간 및 정렬도가 매우 훌륭하여 데이터 기반의 건강 대시보드에 최적입니다.
    - 웹 접근성을 향상시키며 가독성이 높습니다.
*   **적용 방법 (Vite/HTML)**:
    - HTML `<head>`에 Google Fonts 또는 Spoqa CDN 스타일시트를 주입하여 로드합니다.
    ```html
    <link href="//spoqa.github.io/spoqa-han-sans/css/SpoqaHanSansNeo.css" rel="stylesheet" type="text/css">
    ```
    - Tailwind CSS 설정(`tailwind.config.js`)의 `fontFamily`에 `'Spoqa Han Sans Neo', 'sans-serif'`를 선언하여 기본 폰트로 상속시킵니다.

## 3. 국내 모바일 UI/UX 표준 준수 사항

국내의 대형 모바일 서비스(토스, 카카오, 네이버 등) 디자인 시스템의 편의성 기준을 바탕으로 설계 규칙을 정립합니다.

### 1) 엄지손가락 영역 (Thumb Zone)과 터치 타겟
- 스마트폰 화면 하단은 한손 조작 시 엄지손가락이 가장 쉽게 닿는 영역입니다. 중요 입력 버튼(데일리 로그 입력 버튼 등)은 화면 하단 영역에 조밀하게 배치합니다.
- 오작동 및 더블 터치를 방지하기 위해, 모든 터치 가능한 요소(버튼, 아이콘 탭)는 **최소 44px × 44px 이상의 터치 타겟 크기**를 유지하고 최소 8px 이상의 간격을 둡니다.

### 2) 모달 다이얼로그 대신 바텀 시트 (Bottom Sheet) 활용
- 화면 중앙에 뜨는 모달 다이얼로그는 모바일 세로 화면에서 닫기 버튼이 손에 닿지 않아 사용자 피로도를 높입니다.
- 유산소, 무산소, 식사 입력 창 등 데이터를 입력하는 핵심 레이아웃은 화면 하단에서 슬라이드 업되는 **바텀 시트** 형태로 구현합니다. 

### 3) 폼 필드 접근성 가이드라인
- 모바일 가상 키보드가 올라올 때 화면 하단의 입력 폼이 가려지지 않도록 스크롤 여유 마진(Focus Scroll Margin)을 충분히 보장합니다.
- 숫자 입력 필드(`몸무게`, `물 섭취량`, `무게`, `횟수` 등)는 활성화 시 모바일에서 **숫자 전용 키보드(Numeric Keypad)**가 즉시 뜨도록 `inputmode="decimal"` 혹은 `type="number"` 속성을 완벽히 적용합니다.

### 4) 실시간 반응형 피드백
- 네트워크 요청(Gemini API 통신, Supabase 저장 등) 발생 시 버튼에 로딩 스피너를 보여주어 중복 클릭을 방지합니다.
- 데이터 입력 또는 성공적으로 데이터가 복구되면 화면 상/하단에 **토스트(Toast)** 팝업을 띄워 유저에게 동작 상태를 명확히 전송합니다.

## 4. 컴포넌트 UI 규칙 (시트·버튼 일관성)

시트(바텀 시트)마다 버튼·선택 칩·리스트가 제각각으로 흩어지는 것을 막기 위해, 아래 규칙을 **공통 컴포넌트로 강제**합니다. 새 시트를 만들거나 기존 시트를 고칠 때 인라인으로 다시 작성하지 말고 반드시 공통 컴포넌트를 사용합니다.

### 4-1) 도메인 포인트색 매핑

각 기능은 소속 도메인의 포인트색(`variant`)을 따릅니다. (색 정의는 §2, `tailwind.config.js`)

| 도메인 | `variant` | 색 |
| :--- | :--- | :--- |
| 운동 (무산소·유산소·스트레칭·신발) | `exercise` | 에메랄드 `#10B981` |
| 수분·수면 | `trust` | 오션 블루 `#3B82F6` |
| 식단·영양제·업로드(AI) | `highlight` | 비타민 앰버 `#F59E0B` |
| 삭제·경고 | `danger` | 로즈 레드 `#EF4444` |
| 공통/유틸 (프로필 등 도메인 없음) | `default` | = primary(`#10B981`) |

> ⚠️ `default`(=`primary`)와 `exercise`는 색이 같지만 **의미가 다릅니다.** 운동 도메인은 반드시 `exercise`를, 도메인 없는 유틸 액션만 `default`를 씁니다.

### 4-2) 6축 규칙

| 축 | 규칙 | 공통 컴포넌트 |
| :--- | :--- | :--- |
| ① 주(主) 액션 색 | 시트의 도메인 포인트색(4-1) | `SubmitButton` |
| ② 라벨 | 리스트에 항목을 더하면 **"추가"**, 레코드 하나를 덮어쓰면 **"저장"**. 도메인 단어("식단 저장" 등) 금지 | `SubmitButton` |
| ③ 로딩/아이콘 | `busy`면 `Loader2`(스피너), 아니면 액션 아이콘(`추가`=＋ / `저장`=✓). **항상 삼항**, `{busy && …}`(유휴 시 아이콘 없음) 금지 | `SubmitButton` |
| ④ 선택(세그먼트) 칩 | 활성=도메인색, 비활성=`secondary`, 크기=`sm` 고정. 균등폭 고정개수는 `grid grid-cols-N`, 가변 라벨은 `flex-wrap` | `ChoiceChips` |
| ⑤ 헤더 토글·뒤로 | 헤더 모드 토글·뒤로 = `ghost`. 뒤로는 **한 곳에서만** 제공(헤더 또는 푸터 중 하나, 중복 금지) | `Button variant="ghost"` |
| ⑥ 푸터 역할 | 푸터 버튼은 **그 시트의 주(主) 액션**이며 도메인색을 따른다. 커밋형이면 **"추가"/"저장"**(`SubmitButton`), 자동·인라인 저장이라 이미 작업이 끝난 시트의 마무리면 **"완료"**(도메인색 `Button`). 단, 아직 동작을 시작하지 않은 채 빠져나가는 **"닫기"/"취소"/"다시 선택"**은 주 액션이 아니므로 `secondary` 중립 | `SubmitButton` / `Button` |
| (삭제) | 삭제는 `ghost` 아이콘 버튼 + `Trash2`(`text-danger`), 아이콘 `h-4 w-4` | `DeleteButton` |
| (리스트 행) | 행 셸은 `rounded-md border bg-background` 통일 | `ListRow` |

### 4-3) 공통 컴포넌트 위치

`src/components/common/` 에 규칙이 한 곳에 정의되어 있습니다. 규칙을 바꾸려면 시트가 아니라 이 컴포넌트를 수정합니다.

- `SubmitButton` — 주 액션(①②③). `variant`(도메인색)·`busy`·`action="add"|"save"` 전달
- `ChoiceChips` — 선택 칩(④). `options`·`value`·`onChange`·`activeVariant`·`className`(레이아웃)
- `DeleteButton` — 삭제 버튼
- `ListRow` — 리스트 항목 행

### 4-4) 의도된 예외

- **수분 시트 리스트** — 섭취량을 알약형(pill) 칩으로 보여주는 별도 패턴이라 `ListRow` 대신 인라인 펄을 유지합니다.
- **보조 액션** — 수면 "지금" 버튼(`outline`), 신발 상세 "이전 데이터 추가"(`secondary`)는 주 액션이 아니므로 도메인색을 쓰지 않습니다.
